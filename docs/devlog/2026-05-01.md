## 残局求解日志：拆分 proven / done 两个指标

之前日志里 `tail_solve=X%(N/M, Yms)` 只有一个比率，等价于"预算内完成 AND |value|≥1"。拆不出"预算够不够"和"证不证得出"。

改动：
- `TailSolveResult` 加 `budget_exceeded`
- `NetMctsStats` 加 `tail_solve_completed`（预算内完成，不要求 |v|≥1）
- `SelfplayEpisodeResult` 加 `tail_solve_completed` 计数器
- pybind 导出新字段；pipeline 汇总后日志格式变为 `tail_solve=proven=X%(n/m)/done=Y%(k/m), Zms`

语义：
- `done` 偏低 → 预算（depth_limit / node_budget / time_limit_ms）需要加大
- `proven` 偏低但 `done` 高 → 搜到底了但大多是 kProvenDraw 或深度裁剪返回 0；增加深度或等模型更强才能改善

## 训练管线补三项经典 AlphaZero 优化

之前 `training/pipeline.py` 的 `train_step` 没有这些：L2 正则、legal mask、梯度裁剪。都加上：

1. **L2 正则**：`Adam → AdamW(weight_decay=1e-4)`。`weight_decay` 可在 `game.json` 的 `training` 节覆盖
2. **Legal mask 训练**：policy logits 在非法位置填 `-1e9`，log_softmax 后再乘 mask 防止 NaN（直接用 `-inf` 会 `0 * -inf = NaN`）。C++ 侧 `SelfplaySample` 加 `legal_mask` 字段，`selfplay_runner` 和 `heuristic_runner` 都填，pybind 导出
3. **梯度裁剪**：`clip_grad_norm_(max_norm=1.0)`，`grad_clip_norm` 可在配置覆盖（设为 0 关闭）

Warm start 和主训练循环都应用上面三项。

## v9 训练：三个关键修复

v6/v7/v8 runs all showed 0% free eval win rate. Root cause analysis found three issues:

### 1. Feature encoding bug (CRITICAL)
`pipeline.py` called `encode_state(game_id, seed + sample["ply"])` which always encoded the **initial position** with a different seed, not the actual mid-game board state. Every training sample had incorrect features — the model was learning from noise. Fix: use `sample["features"]` directly from C++ selfplay runner, which encodes the correct position at each ply.

### 2. Action space imbalance in heuristic
In typical Quoridor positions: ~3 legal pawn moves vs ~128 legal wall placements. The heuristic scored "advance pawn" and "place gap-improving wall" identically. The uniform tiebreaker meant ~97% of sampled heuristic actions were walls. Fix: +0.1 bonus to pawn moves in `heuristic_pick_quoridor`, making the heuristic prefer advancing.

### 3. Scaffold parameters not wired through
`pipeline.py` wasn't passing tail_solve, temperature_schedule, heuristic_guidance_ratio, or training_filter_ratio to C++. All the scaffold machinery configured in `game.json` was dead code. Fix: complete rewrite of `pipeline.py` to pass all config parameters, plus proper warm start using C++ heuristic episodes.

Config for v9:
- heuristic_guidance: 50% initial, decays over 3000 steps
- training_filter: 80% initial, decays over 3000 steps
- tail_solve: enabled, triggered when either pawn ≤4 steps from goal
- warm start: 800 heuristic episodes at temperature 3.0, 10 epochs

## BUG-008 修复：Splendor temperature_schedule

Splendor 的 `game.json` 用嵌套结构 `temperature_schedule: {initial, final, decay_plies}`，但 pipeline 读平坦键 `temperature_initial` 等。平坦键不存在，默认值 -1.0 被传入 C++，温度衰减静默失效。

修复：添加 `_get_temperature_key()` 辅助函数，先查平坦键再回退到嵌套结构。两种格式都能正确读取。

## BUG-010 修复：pipeline 重写丢失 Replay Buffer

BUG-007 修复重写了 `pipeline.py` 的样本收集逻辑，丢失了 replay buffer（`deque(maxlen=100000)`）。每步只用当前步的 ~7000 样本训练后丢弃，模型无法跨步累积学习。

修复：恢复 replay buffer，每步新样本追加到 buffer，训练从 buffer 随机抽 `train_batches_per_step` 个 batch（默认 3，每个 batch_size=2048）。

## BUG-011 修复（致命）：ONNX 未编译，MCTS 使用均匀策略

**这解释了为什么所有训练步骤的模型 eval 结果完全一致。**

`setup.py` 默认 `BOARD_AI_WITH_ONNX=0`，需要手动设置环境变量。未设置时 `OnnxPolicyValueEvaluator` 静默回退到均匀策略——MCTS 搜索完全不使用神经网络，所有模型（随机、warm start、训练 150 步后）行为完全相同。

发现过程：对比 `model_init.onnx`（随机权重）和 `model_warm.onnx`（warm start 训练后）的 constrained eval，10 局全部逐局相同（winner、total_plies 一模一样）。进一步检查发现 .so 文件中存在 "onnx runtime not enabled at build time" 字符串，且无 onnxruntime 动态链接。

修复：
- `setup.py`：自动检测 `/opt/homebrew` 和 `/usr/local` 的 onnxruntime 头文件，无需手动设置环境变量
- `onnx_policy_value_evaluator.cpp`：ONNX 未编译时构造函数抛异常（不再静默降级），`evaluate()` 在 `!ready_` 时返回 false

## BUG-012 修复：model_init.onnx 导出顺序错误

`model_init.onnx` 在 warm start 训练完成后才导出，和 `model_warm.onnx` 完全相同。修复：在 warm start 之前导出 init 模型。
