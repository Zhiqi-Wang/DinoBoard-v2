## Love Letter 游戏实现

完成 Love Letter（情书）卡牌游戏的完整实现，支持 2-4 人。

### 文件

- `games/loveletter/loveletter_state.h/.cpp` — 游戏状态，template `<int NPlayers>`
- `games/loveletter/loveletter_rules.h/.cpp` — 规则引擎，含全部 8 张卡牌效果
- `games/loveletter/loveletter_net_adapter.h/.cpp` — 特征编码 + BeliefTracker v1
- `games/loveletter/loveletter_register.cpp` — 注册 loveletter/2p/3p/4p
- `games/loveletter/config/game.json` — 训练参数
- `games/loveletter/web/` — Web 前端

### 设计决策

- **动作空间**: 47（固定，不随玩家数变化）
  - Guard: 28 (target*7 + guess-2)，其余按 target 编码
- **特征维度**: NPlayers*28 + 12（2p=68, 3p=96, 4p=124）
- **Belief Tracker v2**: 有状态追踪游戏中获取的精确知识
  - `observe_action`: Priest 偷看→记录对手手牌, Baron 平局→记录双方, King 交换→转移知识, Prince 重摸/淘汰/打出已知牌→清除
  - `randomize_unseen`: 已知手牌直接固定（从 unseen pool 扣除），未知的随机采样
  - 随机来源（牌堆组成）从状态推导，与 Azul 同模式
  - Encoder 持有 tracker 指针，编码时查询 known_hand：已知→真实值，未知→全零占位符
- **hand_exposed 特征**: state 维护 `hand_exposed[NPlayers]`（Priest/Baron/King 触发，出牌/重摸/淘汰清除），encoder 编码为 per-player flag
- **NoPeek + Chance Sampling**: enable_chance_sampling=true，cap=10
  - rng_nonce 随每次抽牌变化，detector 自动识别随机转移
- **全量快照 Undo**: 状态极小（~100 bytes/record），直接 snapshot 比增量 undo 简单可靠
- **玩家淘汰**: 首个支持中途淘汰的游戏。advance_turn 跳过已淘汰玩家

### 项目定位

Love Letter 是框架中对"不完美信息 + 短对局 + 频繁信息交换"类游戏的探索。与 Splendor 不同，难点不在随机来源维护，而在通过 belief tracker + encoder 协作将游戏技能产生的精确知识（Priest 偷看、Baron 比较、King 交换）正确传递给网络。

### 训练

v1 训练 `runs/loveletter_001`（feature_dim=68，已不兼容）：
- 3000 步，loss 2.45→1.79，vs random 综合胜率 86.3%
- v2 改动后 feature_dim=70，需重新训练

## Coup（政变）游戏实现

完成 Coup 卡牌游戏的完整实现，支持 2-4 人。框架中最复杂的状态机游戏。

### 文件

- `games/coup/coup_state.h/.cpp` — 游戏状态，CoupData + CoupState template
- `games/coup/coup_rules.h/.cpp` — 11 阶段状态机规则引擎
- `games/coup/coup_net_adapter.h/.cpp` — 特征编码 + BeliefTracker
- `games/coup/coup_register.cpp` — 注册 coup/2p/3p/4p
- `games/coup/config/game.json` — 训练参数
- `games/coup/web/` — Web 前端（行动声明、质疑、反制、亮牌、交换面板）

### 设计决策

- **动作空间**: 32（固定）
  - 声明动作 0-15（Income/ForeignAid/Coup(t)/Tax/Assassinate(t)/Steal(t)/Exchange）
  - 质疑/允许 16-17, 反制 18-22, 亮牌 23-24, 失去影响力 25-26, 还牌 27-31
- **状态机**: 11 个 CoupStage，将多玩家反应阶段展开为单玩家顺序决策
  - DeclareAction → ChallengeAction → ResolveChallengeAction → ChooseLoseInfluence → CounterAction → ChallengeCounter → ResolveChallengeCounter → ChooseLoseInfluenceCounter → ExchangeReturn1/2 → LoseInfluenceFromAction
- **特征维度**: NPlayers*18 + 21（2p=57, 3p=75, 4p=93）
  - Per-player: alive, coins/12, influence_count(2), revealed_cards(5), known_hand(5), is_active/target/blocker/challenger
  - Global: stage_onehot(11), action_type_onehot(7), ply/200, deck_size/15, is_first_player
- **Belief Tracker**: 极简——unseen pool 完全从状态推导（revealed cards + own hand），observe_action 为空操作
  - 质疑防御成功后牌洗回重抽，没有稳定的 known_hand 信息
  - Tracker 主要价值：randomize_unseen 为 IS-MCTS 重建牌堆+对手暗牌
- **关键规则细节**:
  - Assassinate 费用声明时扣除，被质疑/反制不退还
  - coins ≥ 10 强制 Coup
  - ForeignAid 反制顺时针依次询问（复用 challenge_check_index）
  - Exchange 分两步还牌（kExchangeReturn1 → kExchangeReturn2）

## Peek 模式训练增强

实现 `peek_steps` 训练参数：前 N 个训练步用 peek（全知搜索，禁用 NoPeek traversal limiter），之后自动切回 nopeek。

### 改动
- **`bindings/py_engine.cpp`**: `run_selfplay_episode_py` 新增 `nopeek_enabled` 参数，gate limiter_factory 创建
- **`training/pipeline.py`**: 读取 `peek_steps` 配置，`step <= peek_steps` 时传 `nopeek_enabled=False`
- **`games/*/config/game.json`**: Splendor、Azul、Coup、Love Letter 训练区加 `"peek_steps": 0`
- Arena/eval 不受影响，始终 nopeek

### 设计决策
- 粒度为训练步（不是 episode 内的 ply），与 `heuristic_guidance_steps` 等调度参数一致
- 默认 `peek_steps=0` 向后兼容
- 不做 peek/nopeek 渐变混合（每步要么全 peek 要么全 nopeek），简单可预测

## 残局求解器多人支持 + 仅 Proven Win

### 改动
- **`engine/search/net_mcts.cpp`**: tail solve 结果判定从 `|value| >= 1.0`（proven win 或 loss）改为仅 `kProvenWin`
- Alpha-beta 本身已使用 paranoid 假设（所有对手联合针对当前玩家），天然支持 N 人
- 文档移除"仅双人零和"限制，说明 paranoid 假设和 proven-win-only 策略

### 设计决策
- **仅 proven win 替代 MCTS**: proven loss 在多人场景下过于悲观——对手之间有利益冲突，不一定联合针对你。双人场景下 proven loss 理论上是准确的，但统一行为更简单，且 proven loss 时 MCTS 搜索本身也能找到最优延迟策略
- 不需要修改 `AlphaBetaTailSolver::solve()`，它已经正确用 paranoid minimax 处理 N 人

## 第二轮深度审计与修复

派了三个 agent 并行扫 C++ 引擎、Python 平台层、文档一致性，找到 9 个真正有训练/生产风险的深埋 bug 并全部修复（外加文档清理）。

### HIGH 严重：会毒化训练或污染生产的 bug

1. **Web 找不到训练好的模型，静默退回随机网络**（`platform/game_service/sessions.py:63,127`）
   - 训练输出 `runs/<name>/models/model_best.onnx`，web 服务器读 `games/<id>/model/best_model.onnx`，文件名顺序不同
   - 非 2p 变体找不到模型时 `_ensure_random_model` 静默生成随机权重 ONNX 当成真模型用
   - **修复**: `find_model_path` 两个文件名都接受；缺模型一律抛 `FileNotFoundError`（无论 2p 还是多人），删除 `_ensure_random_model`
   - 2p/多人变体之间如果元数据一致，允许共用基础模型（但要显式 fallback，不静默造新的）

2. **NoPeek limiter 用硬编码种子跑所有 episode**（`bindings/py_engine.cpp:180,258`）
   - `NoPeekTraversalLimiter` 默认种子 `0xB19F3A5D7E991233ULL`，工厂 lambda 没传 `episode_seed`
   - Love Letter / Coup 每个 episode 在同一状态下采样到的对手手牌都一样，belief 分布退化为单点
   - **修复**: 从 `episode_seed ^ 0x9E3779B97F4A7C15ULL` 派生，确保每个 episode ISMCTS 采样独立

3. **Tail solver + belief_tracker 组合可能偷看真实状态**（设计防呆缺失）
   - 当前只有 Azul/Splendor 实际 override 了 `do_action_deterministic`，但接口允许回退到 `do_action_fast`
   - `do_action_fast` 在 Love Letter/Coup 里会从真实 deck 抽牌，tail solver 因此能偷看真实状态
   - 目前 Love Letter/Coup 没注册 tail_solver 所以没触发，但接口层面没有防呆
   - **修复**: 在 `GameBundle` 加 `stochastic_tail_solve_safe` 标志；`GameRegistry::create_game()` 检测到 belief_tracker + tail_solver 而没显式声明 safe 时抛异常；Splendor register 显式设 `true`
   - 违反 CLAUDE.md "AI pipeline 不读取真实状态" 的设计原则，必须硬防守

4. **Web HTTP handler 和 pipeline worker 数据竞争**（`platform/game_service/routes.py:97-106`）
   - `apply_action` 调 `signal_cancel()`（**非阻塞**，只翻 phase 标志）就立刻 mutate `sess["game_session"]` 和 `action_history`
   - pipeline worker 如果在 MCTS 中间，cancel 检查点错过了就会并发写同一个 session
   - **修复**: `apply_action` 改调 `cancel_pipeline()`（阻塞等 worker 释放 lock）
   - 配合 #5 的重构，lock 只在 commit 阶段短暂持有

5. **pipeline_lock 覆盖整个 MCTS 搜索**（`platform/game_service/pipeline.py:189-257`）
   - worker 一开始就拿 lock，整个 5000-sim MCTS 都在 lock 里
   - 用户点 undo 触发 `cancel_pipeline` 时 UI 冻 5 秒以上（违反 CLAUDE.md "locks are session-scoped, held only for state reads/writes"）
   - **修复**: 重构两个 worker：重活（_analyze_user_move、_get_ai_move）在 lock 外跑，最后 `_commit_ai_move` 在 lock 内原子提交；commit 前再检查一次 cancel

6. **`sample.get("auxiliary_score", 0.0)` 等 silent degradation**（`training/pipeline.py:338,456,466-468`）
   - C++ 总是写这些字段，但 Python 用 `.get(default)` 读
   - 如果哪天 C++ 回归没写，所有 auxiliary 训练 target 悄悄变 0（BUG-011 同类模式）
   - **修复**: 全部改成 `sample["key"]` 直接访问；`_human_wr_from_stats` 也去掉 `.get()` fallback

7. **`num_players` 从 JSON fallback 到 `players.max` 再 fallback 到 2**（`training/model.py:39`, `training/pipeline.py:287`）
   - CLI 路径正常注入，但任何绕过 CLI 的调用（测试/脚本）会静默退到 2p
   - value head 维度从 N 维变 2 维，N 维 training target 被 broadcast，loss 算全错
   - **修复**: `num_players` 缺失直接 `KeyError`，要求必须从 `game_metadata()` 注入

8. **MCTS Dirichlet 噪声用 wall-clock 当种子**（`engine/search/net_mcts.cpp:211-216`）
   - `search_root` 没接 seed 参数，Dirichlet 用 `steady_clock::now()` 派生
   - 训练不可复现，同样的 seed 跑两次得到不同的 MCTS 轨迹
   - **修复**: `search_root` 加 `seed` 参数（default 0 保持 wall-clock 为交互场景兜底）；selfplay/arena/其他调用点传 `episode_seed ^ (ply * kGoldenRatio64)` 派生

9. **Tail solver TT key 不含 current_player**（`engine/search/tail_solver.cpp:68,195`）
   - paranoid search 同一个状态 hash 可能对应不同 maximizer/minimizer 节点（取决于当前玩家）
   - 如果游戏 `state_hash(false)` 碰巧不编码 current_player（接口未强制），TT 条目会毒化跨角色节点
   - **修复**: TT key 防御性地 XOR `current_player * kGoldenRatio64`，不依赖游戏作者

### 附带修复

- **`search_root` 平票偏向低编号动作**（`engine/search/net_mcts.cpp:416-423`）：改为在 visit_count 相同的 edge 里用 seeded RNG 随机挑，消除 arena/eval 的系统偏差

## AI API 服务（MVP）

在 `platform/ai_service/` 新增一个观察驱动的 AI 推理 API。动机是用**接口隔离**硬性证明 AI 决策链路不读真实状态——外部调用者只能传 (game_id, seed, my_seat) 和 action_id 观察序列，API 只返回动作和元数据。任何状态字段（board/deck/hand）都不会跨越边界。

### 端点
- `POST /ai/sessions` — 创建 session
- `POST /ai/sessions/{id}/observe` — 提交一个观察到的动作
- `POST /ai/sessions/{id}/decide` — 请求 AI 决策
- `GET /ai/sessions/{id}` — 状态查询（仅返回 is_terminal、current_player、winner、观察计数）
- `DELETE /ai/sessions/{id}` — 关闭 session

### 验证测试（`tests/test_ai_api_separation.py`）
11 个测试：
- 端到端跑完 TicTacToe / Quoridor（确定性游戏），AI 只通过 HTTP 拿动作序列也能玩
- 独立 seed 测试：ground truth 和 AI 用不同 seed。确定性游戏的公共状态与 seed 无关，AI 不读真实状态也应能正常对局——通过
- 契约强制：未知 game_id / seat 越界 / 非法 observe 动作 / 非 AI 回合调 decide 都正确拒绝
- **响应面扫描**：断言任何响应 body 里都不出现 `board/deck/hand/features/legal_mask/state` 等状态关键字，硬防回归
- Session 生命周期管理

### MVP 限制
随机游戏（Splendor / Love Letter / Coup / Azul）目前要求 API 和 ground truth 用同 seed，内部隐藏状态才对齐。未来 v2 改用 per-game public-event 协议（例如 "player 0 flipped card_id=42"），API 内部维持独立 belief。

### 后续用途
除了验证分离，这套 API 本身就是"AI 作为外部服务"的雏形——未来数字化桌游团队可以自己运行游戏服务器，把 AI 决策外包给这个 API，不需要把游戏逻辑集成到框架里。

## Public-Event 协议 + Belief 等价验证

MVP 的 shared-seed 模式其实没证明真正的分离——ground truth 和 AI 内部 state 完全同步。真正的分离是用户提出来的核心思想："同一个游戏模拟状态，我们自博弈内部 AI 的 tracker 是否和服务上的一致"。两者用**独立 seed**，只通过公共事件同步，belief tracker 逐步比对。

### 基础设施（Phase 0）

- `IBeliefTracker::serialize() → AnyMap` 虚方法，输出 canonical 可对比字典
- `GameBundle` 新增四个回调：`public_event_extractor` / `public_event_applier` / `initial_observation_extractor` / `initial_observation_applier`
- `EventPhase { kPreAction, kPostAction }` + `PublicEventTrace { pre_events, post_events }` 类型
- pybind11 新增 `GameSession.apply_observation(action, pre_events, post_events)`（原子 pre → action → post → observe 序列）、`apply_event(phase, kind, payload)`、`apply_initial_observation(perspective, obs)`、`get_belief_snapshot()`
- `run_selfplay_episode` 新增 `trace_perspective` 参数；需要传入**独立**的 `trace_belief_tracker`（因为主 tracker 每步都被 MCTS 重置为 current_player 视角，无法保持稳定 perspective）
- 输出新增 `initial_observation` / `initial_belief_snapshot` / `observation_trace[]`

### 各游戏实现

| 游戏 | Tracker | 事件类型 | 特殊处理 |
|------|---------|---------|---------|
| Azul | stateless | `factory_refill` (post) | 最简单，只需同步 factories + 重算 bag |
| Splendor | `seen_cards` set | `deck_flip` (post), `self_reserve_deck` (post) | mutate_persistent helper 处理 shared_ptr 持久化 state |
| Love Letter | `known_hand` array | `hand_override` (pre/post), `drawn_override` (post) | pre-event 最多——Baron/Guard/Priest/Prince/King 都读对手手牌；初始观察需要包含 `my_drawn_card`（起手玩家已经抽牌了） |
| Coup | stateless | （留作 future work） | Ambassador 的 draw-2-return-2 + 挑战揭露最复杂，其他三个游戏已覆盖所有事件模式 |

### 关键设计决策

1. **`apply_observation` 必须是原子方法**：用户驱动 pre → action → post 后，belief tracker 的 `observe_action` 必须看到 POST-EVENT 的 state_after，否则记下的是 AI 自己的随机结果。早期分开暴露 `apply_event` + `apply_action` 导致 Splendor seen_cards 偏离。

2. **独立 trace_belief_tracker**：selfplay 里的 primary tracker 每步都被 `init(state, current_player)` 覆盖，不能用来追踪稳定 perspective 的 belief。必须从同一个 game bundle factory 再建一个 tracker，专门给 trace 用。

3. **可见性由 ground truth 侧过滤**：`public_event_extractor` 决定给 perspective 看什么。框架不做 firewall——partner 愿意多传也行（对接友好）。

### 测试强度

`tests/test_api_belief_matches_selfplay.py` 对 Azul / Splendor / Love Letter 各跑三个断言（共 9 测试全过）：
- **belief 等价**: 每步 serialize() 相等
- **公开 state 等价**: 终局可见字段逐个相等
- **legal_actions 等价**: perspective 回合的动作集相等（对手回合允许发散，因为对手手牌对 AI 是未知的）

这三条加起来就硬性证明了 AI 决策链路真的不读 ground truth state：任何窃取行为都会让其中一条断言失败。
